# Artifacts Feature — Implementation Plan

> **Status:** Draft — under discussion
> **Created:** 2025-06-23

---

## Summary

Add an **Artifacts** module to Otto that lets users create AI-generated documents (HTML pages, dashboards, reports, prototypes, etc.) stored per-project, browsable from a dedicated sidebar section, and openable as workspace tabs alongside agents, terminals, and browsers.

---

## What is an Artifact?

An artifact is a **single self-contained file** generated by an LLM, stored under a project's `.otto/artifacts/` directory. It is conceptually similar to Claude Code's Artifacts — the model produces a document (typically HTML with inline CSS/JS, or Markdown) that can be previewed live.

Key properties:

- **Single file** — one artifact = one file, not a collection
- **Project-scoped** — stored per project, not globally
- **LLM-generated** — created by prompting a model with user-provided instructions
- **Previewable** — rendered as a live tab in the workspace
- **Persistent** — survives daemon restarts, stored on disk

---

## What the User Asked For

1. New **"Artifacts"** entry in the left sidebar (top menu, alongside Sessions, Schedules)
2. Artifacts page showing a **grid of preview cards** for all artifacts across all projects
3. **Filter by project** via a dropdown next to a "New Artifact" button
4. **Create dialog** similar to the chat message box: name, description, plus all composer controls (provider, model, mode, thinking/reasoning, attachments, documents) — but **no live mode**
5. **Loading state** — spinner on the card while the document is being generated
6. **Card actions** — `...` menu with Delete; Star toggle (favorite) to pin to top
7. **"Open Artifact"** button in the workspace toolbar (next to New Browser / New Terminal) showing artifacts for the current project
8. **Artifact tabs** — new tab type (like browser tabs but without navigation controls), shows artifact content, repositionable, stays up-to-date with file changes
9. **"Create Artifact"** option in the "Open Artifact" context menu, auto-opens the new artifact as a tab on success
10. **Investigate Claude Code's approach** to artifact generation for prompt engineering guidance

## What Might Be Missing / Needs Clarification

### 1. Storage Location: `.otto/artifacts/` per project vs `$OTTO_HOME`

The plan says artifacts live under `.otto/artifacts/` inside each project folder. Otto conventionally stores data under `$OTTO_HOME` (e.g., `~/.otto/agents/`, `~/.otto/schedules/`). There are two approaches:

| Approach                            | Pros                                                                                | Cons                                                                             |
| ----------------------------------- | ----------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| **Per-project `.otto/artifacts/`**  | Co-located with project code; survives Otto uninstall; natural for git (if desired) | Requires scanning project directories; `.otto/` must not be committed by default |
| **Central `$OTTO_HOME/artifacts/`** | Follows Otto convention; single source of truth; easier daemon management           | Artifacts decoupled from project location                                        |

**Recommendation:** Per-project `.otto/artifacts/` aligns with the user's intent and keeps artifacts discoverable alongside project files. The daemon will need to scan known project roots to discover artifacts. Add `.otto/` to a default `.gitignore` suggestion.

### 2. Artifact Content Types

What file formats can artifacts be? Initial support should cover:

- **HTML** (primary) — self-contained HTML with inline CSS/JS, renderable in a browser/webview
- **Markdown** — rendered as formatted text
- **JSON/YAML** — displayed as formatted data (future)

**Recommendation:** Start with HTML-only. The generation prompt instructs the model to produce a single self-contained HTML file. Add Markdown support later if needed.

### 3. Artifact Generation Mechanism

How does the LLM actually produce the artifact? Options:

| Approach                | Description                                                                                                                                 |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **Dedicated agent run** | Spawn a short-lived agent with a system prompt engineered to produce an HTML artifact. The agent writes the file to disk.                   |
| **Direct LLM call**     | Bypass the agent lifecycle; make a single completion call with the artifact system prompt + user prompt, parse the response, write to disk. |
| **Agent tool**          | Register an `artifact_create` tool that agents can call, which handles the file writing.                                                    |

**Recommendation:** Dedicated agent run (short-lived, auto-archived). This reuses the existing agent lifecycle, provider abstraction, streaming, and error handling. The agent gets a system prompt like:

> "You are an artifact generator. Produce a single, self-contained HTML file. The user will describe what they want. Output only the HTML — no explanation, no markdown wrapping."

The daemon watches for the agent to write the file, then signals completion.

### 4. Live Updates ("stays up to date when the document is updated")

The user wants artifact tabs to reflect file changes. This needs:

- A **file watcher** (FS watcher) on the artifact file path
- When the file changes, **push an update** to connected clients via WebSocket
- The artifact tab **re-renders** with the new content

**Recommendation:** Use `fs.watch` or `chokidar` in the daemon to watch artifact files. On change, emit a `artifact.updated` WebSocket event containing the new content (or a hash/ETag for change detection).

### 5. Preview Card Thumbnails

The grid shows "previews" of artifacts. Options:

| Approach                  | Description                                                                    |
| ------------------------- | ------------------------------------------------------------------------------ |
| **Headless render**       | Use Playwright/Puppeteer to render the HTML and capture a screenshot thumbnail |
| **Cached screenshot**     | Store a screenshot alongside the artifact, regenerated on update               |
| **First-content excerpt** | Parse the HTML and show a text excerpt or `<title>` tag                        |
| **Live iframe**           | Embed the HTML directly in a small iframe per card                             |

**Recommendation:** Start with `<title>` + file type icon + creation date as the card content. Add headless screenshot thumbnails as a Phase 2 enhancement (reuses the existing browser-tools/Playwright infrastructure in the daemon).

### 6. Star/Favorites Persistence

Where are starred artifacts stored?

| Approach        | Description                                                                     |
| --------------- | ------------------------------------------------------------------------------- |
| **Client-side** | Stored in the app's local state (Zustand store, persisted to MMKV/AsyncStorage) |
| **Server-side** | Stored in the artifact's JSON metadata on the daemon                            |

**Recommendation:** Server-side, stored as a `starred` boolean field in the artifact's metadata JSON. This way favorites sync across devices (mobile, desktop, web) connected to the same daemon.

### 7. Artifact Metadata

Each artifact needs a companion metadata file (or embedded JSON frontmatter). Proposed schema:

```json
{
  "id": "art_abc123",
  "name": "Dashboard Prototype",
  "description": "A sales dashboard with charts",
  "projectId": "proj_xyz",
  "filePath": "/path/to/project/.otto/artifacts/dashboard.html",
  "kind": "html",
  "starred": false,
  "status": "ready",
  "createdAt": "2025-06-23T12:00:00Z",
  "updatedAt": "2025-06-23T12:05:00Z",
  "generationAgentId": "agent_abc",
  "generationProvider": "claude",
  "generationModel": "claude-sonnet-4-20250514"
}
```

**Recommendation:** Store metadata as a `.json` sibling file next to the artifact (e.g., `dashboard.html` + `dashboard.html.json`). This follows the agent record pattern (`agentId.json`) and keeps things file-based.

### 8. Artifact Tab Type

A new workspace tab `kind: "artifact"` needs to be registered in the panel system. It should:

- Show the artifact content (HTML rendered in a webview/iframe, or Markdown rendered as text)
- Have no navigation controls (no URL bar, no back/forward)
- Subscribe to file-change events for live updates
- Show the artifact name in the tab header
- Support reordering/repositioning like other tabs

### 9. Error States

What happens when:

- Generation fails (model error, timeout)?
- The artifact file is deleted externally?
- The project is removed?
- The HTML is malformed?

**Recommendation:**

- Failed generation: show error on the card, allow retry or delete
- Missing file: show "File not found" state, allow delete (orphan cleanup)
- Missing project: gray out or hide the artifact card
- Malformed HTML: show raw content with a warning banner

### 10. Security / Sandbox

Artifacts are HTML files rendered in the workspace. They could contain scripts.

**Recommendation:** Since artifacts are user-generated and rendered locally, treat them as trusted content (same security model as browser tabs). However, consider adding a note in the UI that artifacts run locally and may execute JavaScript.

### 11. i18n

All UI strings need translation keys following the existing `i18n/` convention.

### 12. Capability Flag

Following the codebase convention, this feature should be gated behind a capability flag in `server_info.features.artifacts` with a `COMPAT(artifacts)` comment.

## Architecture

### Data Flow

```
User clicks "New Artifact"
  → Opens creation dialog (name, description, provider, model, mode, thinking, attachments)
  → Submits to daemon via WebSocket RPC: artifact.create.request
  → Daemon creates a short-lived agent with artifact system prompt
  → Agent generates HTML, writes to .otto/artifacts/{id}.html
  → Daemon detects file write, updates metadata to status: "ready"
  → Daemon emits artifact.updated event to all connected clients
  → Client updates the artifact card (spinner → preview)
  → User can open the artifact as a tab in any workspace for that project
```

### WebSocket RPCs (following `docs/rpc-namespacing.md`)

| Direction    | Name                                   | Purpose                                                                 |
| ------------ | -------------------------------------- | ----------------------------------------------------------------------- |
| Request      | `artifact.client.list.request`         | List artifacts, optionally filtered by project                          |
| Response     | `artifact.client.list.response`        | Array of artifact metadata                                              |
| Request      | `artifact.client.create.request`       | Create a new artifact (name, description, provider config, attachments) |
| Response     | `artifact.client.create.response`      | Artifact ID and initial metadata                                        |
| Request      | `artifact.client.delete.request`       | Delete an artifact                                                      |
| Response     | `artifact.client.delete.response`      | Success/failure                                                         |
| Request      | `artifact.client.star.request`         | Toggle starred status                                                   |
| Response     | `artifact.client.star.response`        | Updated metadata                                                        |
| Request      | `artifact.client.get-content.request`  | Get the file content for an artifact                                    |
| Response     | `artifact.client.get-content.response` | File content string                                                     |
| Notification | `artifact.daemon.updated.notification` | Push: artifact file changed or generation completed                     |
| Notification | `artifact.daemon.created.notification` | Push: new artifact appeared                                             |
| Notification | `artifact.daemon.deleted.notification` | Push: artifact was removed                                              |

## Claude Code Artifacts Investigation

### How Claude Code Does It (Known Behavior)

From public observation and documentation:

1. **Detection:** Claude detects when it should produce an artifact vs. regular chat output. This is triggered by certain user intents (build a UI, create a page, make a diagram) or explicit `/artifact` commands.

2. **System Prompt:** Claude uses a special system prompt that instructs the model to produce self-contained, renderable content. The prompt emphasizes:
   - Single file output
   - Self-contained (inline CSS/JS, no external dependencies)
   - Valid, working code
   - No markdown wrapping around the code

3. **Content Types:** HTML, React, SVG, Mermaid, Markdown, code snippets.

4. **Streaming:** The artifact content streams in real-time and renders progressively.

5. **Iteration:** Users can ask for modifications, and Claude updates the artifact in-place.

### What We Need to Adapt

For Otto's implementation with OpenAI-compatible and frontier models:

1. **Artifact System Prompt** — A reusable prompt template that instructs any model to produce a single self-contained HTML file. This needs to work across providers (Claude, GPT, Codex, etc.).

2. **Output Parsing** — The model might wrap HTML in markdown code blocks, add explanations, or produce partial output. We need robust parsing to extract the HTML content.

3. **Validation** — Check that the output is valid HTML before saving. Handle malformed output gracefully.

4. **Streaming Preview** (Phase 2) — Show the artifact rendering progressively as the model generates, rather than waiting for completion.

### Proposed Artifact System Prompt

```
You are an artifact generator for Otto, a development environment.

Your task is to create a single, self-contained HTML file based on the user's description.

RULES:
- Output ONLY valid HTML. No explanations, no markdown, no code fences.
- The HTML must be completely self-contained: all CSS inline or in <style> tags, all JS in <script> tags.
- Do not reference external resources (CDNs, images, fonts) unless absolutely necessary.
- Use modern, semantic HTML5.
- Make it visually polished and functional.
- If the user describes a complex application, create a working prototype with mock data.
- Handle edge cases gracefully (empty states, loading states, errors).

The user will describe what they want. Produce the complete HTML file.
```

## Viewing the generation log

The artifact generation agent is a **real provider agent** (`agentManager.createAgent` → `runAgent` in `artifact-service.ts`), so it streams normal chat/tool-call events during generation just like any other agent. Users can watch this live: while an artifact is `status: "generating"`, the artifact panel (`packages/app/src/panels/artifact-panel.tsx`) shows a **"View generation log"** button that opens the generation agent as an ordinary `{ kind: "agent", agentId }` workspace tab (the `generationAgentId` is stored on the artifact metadata). This reuses the existing agent panel + timeline fetch/stream infrastructure — no dedicated log UI.

Getting this to work end-to-end required threading past **three** separate places that deliberately hide internal agents. Each is a real gotcha for anyone touching this again:

### 1. `internal` vs `observable` — the stream filter (the non-obvious one)

`internal: true` means the agent is hidden from listings and not persisted. Crucially, it **also** makes `AgentManager.dispatch()` drop the agent's `agent_stream` events for **global subscribers** — and the daemon session holds exactly one global subscription (no `agentId`), through which it forwards all live events to clients. So an `internal` agent's messages never reach the client's live stream. The client's timeline _fetch_ (`fetch_agent_timeline_request`) bypasses this filter, which is why navigating away and back showed messages but they never arrived live.

The fix is a separate opt-in flag: **`observable: true`** (see `AgentSessionConfig` / `ManagedAgent`). An observable-internal agent still has its `agent_stream` forwarded to global subscribers, while its `agent_state` stays filtered (so it never appears in the sidebar/agent list). Artifact agents set both `internal: true` and `observable: true`. Other internal agents (branch-name / git-metadata generators) stay non-observable and fully silent. The client side needs no change — its `agent_stream` handler enqueues by `agentId` and only renders events for an open panel.

Regression test: `agent-manager.test.ts` → "global subscriber receives stream for observable internal agents but not plain internal ones".

### 2. Identifier resolution excludes internal agents

`Session.resolveAgentIdentifier` (used by `fetch_agent_request`, which the agent panel calls on mount) builds its "known agent ids" set from `agentStorage.list().filter(!internal)` **and** `agentManager.listAgents().filter(!internal)` — both exclude internal agents — so it returned "Agent not found" even for a live artifact agent. There's now an exact-id fast path at the top of `resolveAgentIdentifier` that checks `agentManager.getAgent(id)` directly, bypassing the internal-exclusion for callers that already hold the literal `agentId` (the fuzzy prefix/title matching below still excludes internal agents).

### 3. Client-side tab reconciliation prunes "unknown" agent tabs

`workspace-screen.tsx` runs `reconcileWorkspaceTabs`, which closes any open `agent` tab whose id isn't in the workspace's known/active agent set once agents are hydrated. Because the artifact agent is internal (never broadcast via `agent_state`), it's never in that set, so a freshly opened log tab was pruned within the same tick. The fix folds generating artifacts' `generationAgentId`s into the reconciliation snapshot's known/active sets (scoped to the current server + workspace), exempting an explicitly opened log tab from pruning without adding it to the sidebar or any auto-open logic.

### Scope boundary: live-only

This is **live viewing only**. When generation finishes, `spawnArtifactAgent`'s `finally` block calls `agentManager.closeAgent(agent.id)`, which drops the in-memory timeline immediately; since the agent is internal it was never persisted either. So reopening the log after completion correctly shows the agent-panel "not found" state. Making a completed generation's transcript durable (delay `closeAgent`, or persist a lightweight snapshot) is deferred future work.

## Open Questions

1. **Should artifacts be editable by the user directly?** (Not just regenerated) — Out of scope for now, but worth considering.

2. **Should artifacts be shareable/exportable?** (Download the HTML file) — Likely yes, add to the `...` menu.

3. **Should there be an artifact size limit?** — To prevent the model from generating enormous files. Suggest 1MB cap.

4. **Should artifacts support multiple files?** (e.g., HTML + companion CSS/JS) — The plan says single file. Keep it simple for now.

5. **How do we handle artifact versioning?** — If a user regenerates an artifact, do we keep history? — Out of scope for now.

6. **Should the artifact agent be visible in the agent list?** — It is marked `internal: true` so it doesn't clutter the sidebar, **but also `observable: true`** so its live chat can still be watched via the generation log. See "Viewing the generation log" below.

7. **What happens to artifacts when a project is removed from Otto?** — They should be hidden (not deleted) and shown with a "project not found" state.

## Risks & Mitigations

| Risk                                | Mitigation                                                                                              |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------- |
| Model produces invalid HTML         | Parse and validate before saving; show error state on failure                                           |
| Generation takes too long           | Set a timeout (e.g., 2 minutes); show timeout error                                                     |
| File watcher misses changes         | Use a polling fallback; hash-based change detection                                                     |
| Large artifacts slow down the grid  | Lazy-load card previews; virtualize the grid                                                            |
| Cross-provider prompt inconsistency | Test the system prompt against multiple providers; maintain provider-specific prompt variants if needed |
| Storage bloat from many artifacts   | Add a per-project artifact count limit or total size limit                                              |
